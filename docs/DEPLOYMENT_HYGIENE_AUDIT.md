## Section E — Risk Assessment

| Recommendation | Risk | Justification |
|---|---|---|
| Add `scratch/` to `.gitignore` and untrack | **LOW** | All 49 files are self-contained investigation scripts and their outputs. Zero application imports. Zero build references. The auth migration work they validated is complete and shipped. Worst case: someone wants to re-read an old benchmark result — they can check git history. |
| Untrack `next-env.d.ts` and `tsconfig.tsbuildinfo` | **LOW** | Both are already in `.gitignore`. They were committed erroneously. Removing them from tracking is the correct state; Next.js and TypeScript regenerate them automatically. |
| Delete `src/lib/middleware.ts` | **LOW** | Confirmed zero imports. The file is a copy of Supabase SSR boilerplate that was never wired up. The real middleware (`src/middleware.ts`) is independent. No functional impact. |
| Delete `src/lib/client.ts` and `src/lib/server.ts` | **LOW** | Both are confirmed duplicate factories never imported by any file. The canonical versions under `src/lib/supabase/` are the ones in use. Deleting the duplicates prevents future confusion about which client to import. |
| Delete `src/lib/auth-utils.ts` | **LOW** | `checkEmailExists()` was never integrated into any auth flow. Zero imports. Safe to delete. If email existence probing is ever needed, it should be re-evaluated against current security posture (email enumeration risk) before re-introducing. |
| Delete root `favicon.ico` | **LOW** | Duplicate of `public/favicon.ico`. Next.js only serves from `public/`. The root-level copy is never requested. Removing it eliminates an ambiguous file. |
| Keep `migrations.sql` and `supabase_setup.sql` | **N/A — keep** | These are the authoritative database schema references. Losing them would make environment recreation significantly harder. They must remain version controlled. |
| Keep `vercel.json` | **N/A — keep** | Removing this would break the `/api/keep-alive` cron job in production, causing the Supabase project to go idle on the free tier. |

**Aggregate deployment risk of acting on all recommendations:** LOW. No application code paths, build outputs, or deployment configurations are affected. All deletions are confirmed-dead files. The `.gitignore` changes only affect version tracking, not the running application.
